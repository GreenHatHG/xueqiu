# 雪球抓取：数据流转、抓取方式与登录要求（实测记录）

本文档用于把“数据从哪里来、怎么串起来、怎么落盘、哪些必须登录、以及有哪些必须注意的反爬/账号安全点”先记录清楚，避免后续写代码时边写边猜。以下结论以 **2026-03-07** 的登录态实测为准；雪球接口与风控策略可能随时调整，因此实现时以“保守、可停止、可恢复”为原则。

## 1. 总体原则：只走浏览器内会话，不走裸 HTTP

雪球存在 WAF/风控与签名机制，不同请求路径触发的挑战强度也可能不同。因此当前实现的总原则是：**所有取数都尽量留在已登录浏览器会话内部完成**。但是“留在浏览器里”并不等于“所有数据都必须靠 UI 滚动触发”。随着实测推进，当前代码已经收敛成一套分类型策略：

第一，`timeline`（用户时间线）当前仍然固定走 UI 页面驱动：打开用户主页、滚动页面、必要时点“下一页”，然后拦截页面自己发出的 `/v4/statuses/user_timeline.json`。这次只是补了进度记录，让它中途停掉后能先快进回到上次附近，再继续往后抓。

第二，`comments`（用户全站回复）当前主流程会优先在浏览器会话里直接请求 `/statuses/user/comments.json` 并按 `max_id` 游标翻页。这里说的“浏览器会话”不是让你手工先开一个 Chrome 再连进去，而是程序先准备一份基础浏览器资料目录，再为每个用户复制一份新的目录并起一个新的 Chrome。这样既能尽量保留真实浏览器会话的稳定性，也能把用户和用户之间的脏状态隔开。

第三，`talks`、详情页补全文这类上下文补齐，当前仍然沿用现有的 API/detail 标签页路径，不跟 timeline/comments 的主抓取通道绑死。

因此更准确的说法已经不是“方案固定为 UI 页面驱动”，而是“方案固定为浏览器内会话驱动”；UI 拦截和浏览器会话内 direct fetch 都属于这个原则的具体实现。

## 2. 数据类型与“主干—分叉”的流转结构

我们要的内容分成三类，但它们之间不是并列关系，而是有一条最稳定的“主干”与若干“上下文分叉”：

第一类是“博主自己发的内容”（status 时间线）。它能覆盖博主发布的动态，但通常 **覆盖不了** 博主在全站的所有回复链，因此它更像“作品集”，不是“评论全量”。

第二类是“博主在全站的回复/评论”（个人主页的“回复”Tab）。这条数据流是主干：它天然以“博主说过的话”为中心，并且每条回复 item 里带有 `status`（原博文/动态）的关键信息，使得我们能够把“这句回复”准确定位到“它发生在哪一条原帖下”。这正好解决你提到的痛点：回复列表里如果只看到问答片段，不知道具体属于哪条博文，就无法还原上下文。

第三类是“上下文补齐”。它分两种：

1）“查看对话”（talks）：用于还原该回复所在的对话链（可能很长）。
2）“评论区子回复”（child replies）：用于还原“某条评论下面的回复串”，尤其是 UI 上显示“查看 N 条回复”的那种折叠链路。

在实现上建议以“博主全站回复列表”为驱动：每抓到一条博主回复，就按需派生抓取 talks/子回复/评论区主列表，并按唯一键去重与缓存，避免重复打接口。

## 3. 关键接口（按数据流顺序）

### 3.1 博主发言：用户时间线（statuses）

接口（已登录浏览器内打开接口 URL）：
`GET https://xueqiu.com/v4/statuses/user_timeline.json?page={n}&user_id={uid}`

用途：抓取博主自己发的动态（按时间向旧翻页）。它更适合作为“发布内容归档”，也能为后续做“回复归属到哪条 status”提供补充对照（但不是必须依赖）。

分页：`page=1,2,3...`；当某页返回空或明显不再变化时停止。实现时建议提供“最多页数/最多条数”的硬阈值，先小跑验证。

当前代码主路径说明：timeline 仍然固定走 UI 页面驱动与网络拦截，不改成主流程 direct fetch。原因也很直接：这条路径在实际环境里更稳，直接请求更容易把验证顶出来。

### 3.2 博主全站回复：用户回复列表（comments）

接口（已登录浏览器内打开接口 URL）：
`GET https://xueqiu.com/statuses/user/comments.json?user_id={uid}&size=20&max_id={cursor}`

用途：抓取博主在全站发出的 comment 列表（这是主干数据源）。

分页与限制：`size` 实测上限为 20；用返回里的 `next_max_id` 做游标向后翻。终止条件建议同时具备三类：返回 `items` 为空、或 `next_max_id=-1`、或游标重复（避免接口偶发异常导致循环）。

当前代码主路径说明：comments 现在会优先直接请求这个接口，并按 `next_max_id` 连续向后翻页。这条路径的优点是不会像 UI 滚动那样在“页面不再主动触发新请求”时提前停住；代价是程序必须先准备好一份能稳定返回 JSON 的基础浏览器资料目录，然后每个用户都用复制出来的新目录单独起浏览器。

“定位原帖”的关键字段（用于解决“只见问答不知原帖”）：每条 item 通常带 `status` 对象。

- 常见情况：`status.user_id != -1`，原帖 URL 可拼为 `https://xueqiu.com/{status.user_id}/{status.id}`。
- 另一类情况：`status.user_id == -1`（常见于标的/讨论页等），原帖路径在 `status.target`，原帖 URL 为 `https://xueqiu.com{status.target}`。

实现时应把“原帖定位信息”作为规范化字段写入落盘（例如 `root_status_id`、`root_status_url`、`root_status_target`），以便后续无需二次计算。

### 3.3 查看对话：talks（对话链上下文）

接口（已登录浏览器内打开接口 URL）：
`GET https://xueqiu.com/statuses/talks.json?id={root_status_id}&comment_id={comment_id}&page={p}&count=50&asc=true`

用途：给定“某条博主回复 comment”，拉取“查看对话”里的整条链路，补齐对话上下文。

分页与限制：`count` 实测单页上限 50；响应包含 `page` 与 `maxPage`。建议按 `page=1..maxPage` 逐页请求；同时容忍 `page>maxPage` 返回空数组这一稳定语义。

### 3.4 原帖评论区主列表：v3 comments（type=1）

接口（已登录页面内 fetch）：
`GET https://xueqiu.com/statuses/v3/comments.json?id={status_id}&type=1&size=20&max_id={cursor}`

用途：在需要“还原原帖评论区结构”或“定位父评论对象”时使用。注意该接口在脱离浏览器 cookie 的情况下更容易触发挑战页，因此实现上仍然只在页面上下文里调用，并且把“返回不是 JSON”视为风控信号。

分页：使用 `next_max_id`。终止同样以“空列表 / next_max_id=-1 / 游标重复 / 本页无新增”组合判断更安全。

### 3.5 某条评论下的子回复：v3 comments（type=4 & child_type=2）

接口（已登录浏览器内打开接口 URL，来自 UI “查看 N 条回复” 行为）：
`GET https://xueqiu.com/statuses/v3/comments.json?comment_id={parent_comment_id}&type=4&child_type=2&id={status_id}&max_id={cursor}&size=20`

用途：当博主回复是“在评论链里回复某条评论”时，我们需要把那条父评论以及其下的回复串补齐，才能还原对话在评论区的上下文。实测返回结构里 `comments[0]` 是父评论对象，子回复在其 `child_comments` 数组里。

注意事项（非常重要）：该接口的 `next_max_id` 在实测中可能出现“`-1` 与某个大数游标反复切换”的异常分页表现。为了避免死循环，必须同时使用以下停止条件：`next_max_id==-1`、或 `next_max_id` 重复、或本页 `child_comments` 无新增 id（用集合去重判定）。

## 4. 哪些需要登录才能查？（以及“怎么判定没登录/被风控”）

从稳定性角度，以上接口都应当视为“需要登录态（cookie/本地存储）才可靠”。即使某些内容在“未登录”理论上也能看到，实际抓取时仍可能被 WAF 以挑战页形式拦截；并且常见情况是“导航打开能返回 JSON，但 XHR 返回挑战页 HTML”。本项目因此以“导航打开接口 URL”为首选取数通道。

这条结论需要补一层当前实现语境。现在并不是所有接口都统一走“导航打开接口 URL”：timeline 仍然主要靠页面驱动和拦截；comments 则已经证明可以在当前浏览器页面上下文里 direct fetch 稳定返回 JSON。因此更准确的经验规则应该是：**优先沿用当前会话里已经验证稳定的那种方式**，而不是死守“导航一定优于 fetch”或“fetch 一定优于导航”这种绝对说法。

实现层面的判定方式也比之前更保守了一点：每次请求都预期返回 JSON；如果返回体是 HTML（例如以 `<!DOCTYPE html>` 开头，或 `content-type` 明显不是 JSON），就把它当作“未登录或被风控”的信号，触发退避、降速与最终停止（而不是继续猛刷）。但如果响应本身已经是可正常解析的 JSON，就不应再因为正文里出现“验证/刷新/滑动”等普通词汇而误报成挑战页。

因此，“登录”不只是为了权限，更是为了拿到可稳定解析的 JSON。我们不会在代码里保存账号密码；只支持你在可见浏览器窗口里手动登录一次，然后用持久化上下文复用登录态。

## 5. 反爬与账号安全注意事项（强制保守）

为了尽量降低封禁风险，抓取时需要遵守以下保守策略（实现时会做成默认行为，而不是可选项）：

第一，单上下文、单页面、串行请求。并发抓取会显著放大风控触发概率，也更难在异常时快速止损。

第二，节流与随机抖动。每次请求之间至少等待一个固定下限，并叠加小幅随机 jitter，让行为更接近人类浏览。

第三，对 429/403、网络错误、以及“返回 HTML 挑战页”做指数退避重试；重试次数达到阈值后自动停止，并把当前游标/进度落盘，确保下次可以从断点恢复，而不是继续撞风控。

第四，本地去重与缓存。对于同一个资源（同一个 `status_id/comment_id/page/max_id`），重复运行时不应重复请求或重复落盘；这既降低访问量，也提升可恢复性。

第五，小步扩大范围。默认只抓最近少量页数做验证，通过后再逐步扩大最大页数/条数，避免一次性全量历史导致长时间高频访问。

## 6. 落盘与可恢复性（这一节更偏设计思路，不是当前真实表结构）

为了支持“分批运行、失败重试、不重复写入”，落盘建议分两层：

第一层是“主表”（SQLite 表）：例如 `statuses`、`comments`，每条记录带有稳定主键（`status_id` 或 `comment_id`），并包含规范化后的核心字段（时间、文本、关联的 root status 定位信息等），同时保留原始 JSON 文本字段便于回溯与补字段。

第二层是“上下文快照”（JSON）：例如 talks 与子回复这种多页聚合的结构，按 `{root_status_id}/{comment_id}` 或 `{status_id}/{parent_comment_id}` 组织成独立文件，便于后续只重拉某个上下文而不影响主表。

实现时还应当维护一份“进度/游标状态”（例如每个接口的最后 `next_max_id` 或最后 page），使得停止后可继续跑，而不是每次从头扫。

上面这一节主要是为了讲“如果只看数据分层，最容易理解的拆法是什么”。但当前仓库的真实实现已经不是按那种多张事实表在跑了，实际请以下一节为准。

## 7. 当前实现补充：`merged_records`、`context_json`、`payload_json` 与 `entry` 语义

上面第 6 节描述的是“按事实层拆表”的建议颗粒度，便于讨论抓取与恢复策略；但当前仓库里的实际实现已经进一步收敛为一张 SQLite 表 `merged_records`。程序先把 `status`、`comment`、`talk` 分别按唯一键写入这张表，最后再折叠成适合直接阅读的 `entry` 记录。因此，后续如果你是为了做“观点抽取”“标签聚合”之类的二次处理，必须先理解这张表里几个字段的分工，不能只盯着 `text`。

另外，当前真实落库不只有 `merged_records`。为了支持断点续跑，程序还会同时维护 `crawl_progress` 和 `talks_progress` 这两张进度表，分别记主流程和“查看对话”补齐跑到哪了。

首先，`context_json` 存的是轻量上下文和关联键，目的是让程序快速知道“这一行是什么、和谁相关、该怎么串起来”。例如 `status` 行会写入 `status_id`、`status_url`，`comment` 行会写入 `comment_id`、`root_status_id`、`root_status_url`，而最终的 `entry` 行则会写入 `entry_type`、`topic_status_id`、`comment_id` 等字段。它更像一张“索引卡片”，适合做筛选、归类和关系判断，也适合在不展开完整原始数据的情况下快速定位一条记录的大致语义。

其次，`payload_json` 存的是完整来源数据和处理后的原始载荷，目的是保留可回溯的事实源。`status` 和 `comment` 的 `payload_json` 通常直接包住原始 `record`；`talk` 则会同时保留 `raw`、`clean` 和分页元信息；最终折叠成 `entry` 后，`payload_json` 里会继续挂住这条展示记录背后的 `status`、`comment`、`talk` 来源。它更像“档案袋”，适合后续重新抽字段、重做归因、重跑标签，而不是拿来做快速列表展示。

换句话说，`text` 是给人看的展示文本，`context_json` 是给程序做快速判断的轻量索引，`payload_json` 才是后续需要追溯事实时最可靠的来源。尤其在做投资观点抽取时，不建议直接把 `text` 当成唯一输入，因为它可能已经是多段内容拼接后的展示结果。

这里再补一句很容易误会的点：`text` 现在也不是“全清洗后的纯文本”。当前代码只会顺手去掉 `<a ...>` 包裹、把 `<br>` 变成换行、把雪球表情图那类 `<img ...>` 换成对应文字；像 `<p>`、`<strong>`、`<figure>`、大部分普通 `<img>` 等标签，仍然可能保留在 `text` 里。所以如果后面要把 `text` 直接拿去做检索、打标签或喂模型，最好再自己补一层正文清洗。

再说 `entry` 里的两种类型。`entry:status:{status_id}` 表示一条独立发言，也就是最终展示层的一条“单条原帖/动态”；它通常对应一个作者自己的完整状态，适合在后续处理中作为一条相对原子的观点来源。`entry:chain:{topic_status_id}:{comment_id}` 表示一条评论链，也就是“围绕某个主题原帖展开的一次评论观察”；其中前半段的 `topic_status_id` 表示这条链归属的主题原帖，后半段的 `comment_id` 表示这次评论本身。

这里最容易踩坑的地方在于：`chain` 不是“单条评论正文”，而是一个为了阅读方便拼接出来的展示结果。当前实现会把原博文、评论正文、以及可能抓到的对话链合并成一条 `text`。因此，`entry:chain` 很适合你肉眼回看上下文，但不适合被直接当成“原子观点”去打标签；如果后续直接按 `chain.text` 抽标签，原博主题会在多条评论链里反复出现，导致标签重复膨胀。更稳的做法是：先通过 `context_json` 判断这是一条 `chain`，再回到 `payload_json` 中只提取真正属于当前观察对象的那段正文，例如优先使用 comment 自己的正文，把 talk 当辅助上下文，而不是把整条拼接文本视为同一层级的话语。

因此，后续如果要做“观点时间线”“风险提醒”“按股票或行业聚合”的派生层，可以把 `status_id` 或 `comment_id` 视为单条来源 id，把 `topic_status_id` 视为主题串联 id；但不要把 `merge_key` 本身当成业务标签，也不要把整条 `chain.text` 直接当成最终事实。这样才能既保留展示可读性，又避免在结构化分析时把同一主题重复计算多次。
